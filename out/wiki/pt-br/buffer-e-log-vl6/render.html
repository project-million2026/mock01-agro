¶ Configurações e Consultas do Buffer FIFO/LIFO e LOG

O VIRLOC 6 possui uma memória para armazenamento de reportes enquanto o equipamento estiver offline (em área sem sinal, por exemplo). Esse espaço é compartilhado com a memória de Log, e pode ser configurado para melhor atender a aplicação. O parâmetro que define essas configurações é o VQ16,2E que será detalhado adiante.

 

¶ ⚙️ Configuração

>VS16,2E,56<

ou

>VS16,2E,57<

Quando o valor do parâmetro destacado for um número par, o buffer está configurado como FIFO (first-in, first-out), a primeira mensagem gerada é a primeira mensagem que sai;

Quando o valor do parâmetro destacado for um número ímpar, o buffer está configurado como LIFO (last-in, first-out), a última mensagem gerada é primeira mensagem que sai.

 

Para configurar a quantidade total de registros, devemos atentar às seguintes regras:

  • Mínimo: 32;
  • Máximo: 160 para placas com micro L48 (BR61LQ07 a BR61LQ11) ou 400 para placas com micro F91 (BR61LQ12, BR60, BR59 ou BR58);
  • O valor definido para o parâmetro deve ser múltiplo de 8;
  • Será multiplicado por 32 para definição da quantidade de registros que estarão disponíveis para buffer de envio.

 

Obs: para as versões de VL06 com memória estendida, o valor máximo do parâmetro pode ser de até 7328. Consultar o time comercial sobre disponibilidade.

 

¶ Exemplos

>VS16,2E,56<

56 x 32 = 1792 registros para buffer de envio FIFO

 

>VS16,2E,57<

57 x 32 = 1824 registros para buffer de envio LIFO

 

Dessa forma, se define a quantidade de registros para buffer, e o restante da memória é alocada para armazenamento dos registros de LOG.

 

¶ ? Consultas

Para consultar a configuração atual do buffer:

>VQ16,2E<

Retorno:

>VR16,2E,80<

Onde 80 é o parâmetro configurado (por padrão, o valor é de 56).

 

Para consultar a quantidade total de registros atualmente alocada para buffer e LOG:

>GFFP<

Retorno:

>GFFP000084,000084,000000,001024,000000,000000,013312,00,00,3,0,0<

Onde:

000084 = localização do ponteiro de escrita FIFO/LIFO;

000084 = localização do ponteiro de entrega com prioridade baixa (_LO) e normal (_OK);

000000 = localização do ponteiro de entrega com prioridade alta (_HI);

001024 = capacidade total de buffer FIFO/LIFO;

000000 = localização do ponteiro de escrita LOG;

000000 = localização do ponteiro de entrega LOG;

013312 = capacidade total de LOG.

 

Para consultar quantos reportes estão na fila FIFO/LIFO atualmente:

>GFFX<

Retorno:

>GFFX,127,0,0<

Onde:

GFFX = Tipo de pacote (resposta de uma consulta GFFX);

127 = quantidade de mensagens em buffer com prioridade normal (OK) a serem enviadas;

0 = quantidade de mensagens em buffer com prioridade alta (HI) a serem enviadas;

0 = Quantidade de mensagens em LOG a serem enviadas.

 

Observação:

Um reporte irá ocupar uma quantidade de registros proporcional ao seu tamanho. Para comparação:

  • Um reporte GP (retorno da consulta QGP) ocupa 2 registros;
  • Um reporte TT (retorno da consulta QTT) ocupa 3 registros;
  • O maior reporte suportado pelo protocolo XVM (125 caracteres) ocupa 8 registros.

Para saber a quantidade de registros ocupada por um reporte, basta verificar o incremento do ponteiro de escrita FIFO/LIFO conforme os reportes são gerados.

 

¶ ? Limpando a Memória

Para apagar os reportes salvos em memória de buffer LIFO/FIFO e LOG, utiliza-se o comando abaixo:

>GFFEpppppppp<

Onde pppppppp é a senha geral do equipamento (padrão: 00000000).

O comando é confirmado após o Virloc retornar:

>GFFE_ACK<

Atenção: após a execução desse comando todos os reportes em fila são descartados e toda a memória de buffer é apagada, não havendo possibilidade de recuperação.

 

¶ Utilizando o LOG interno

O LOG pode ser utilizado para armazenar reportes internamente no Virloc para serem requisitados para descarga posteriormente. Isso permite a criação de funcionalidades como uma “Caixa Preta”, em que os reportes são armazenados em uma frequência alta (a cada segundo, por exemplo) e caso haja algum sinistro, ou após um tempo predefinido, esses dados são extraídos e enviados via serial, Bluetooth ou GPRS.

 

¶ ? Salvando dados no LOG interno

Para salvar um reporte na memória, utiliza-se o destino LOG, conforme o exemplo abaixo:

>SED000 TT00++ IN07++ LOG GP<

No exemplo, a cada segundo com a ignição ligada, será armazenado um reporte do tipo GP no LOG interno. Conforme os registros são salvos, é possível verificar os ponteiros sendo incrementados nas consultas GFFP e GFFX, detalhadas acima.

 

¶ ? Entrega dos Dados do LOG interno

Para descarregar a memória do log, existe o comando GFFN, que solicita todos os últimos registros não entregues. Os reportes são enviados um a um em pacotes separados e não exigem confirmação de ACK; caso seja necessária a confirmação, basta adicionar o modificador _HI.

>GFFN< //Solicita novos registros como prioridade normal (sem ACK para Bluetooth ou serial, com ACK para GPRS)

>GFFN_HI< //Solicita novos registros como prioridade alta (com ACK para todos os canais)

 

¶ Entrega Manual do LOG

Desta forma, os registros são enviados para o mesmo canal por onde o comando foi recebido. Exemplo:

21/10 15:40:03 TX: >GFFN;ID=LN61;#8094;*1F<

21/10 15:40:03 RX: >GFFN_ACK;ID=LN61;#8094;*09<

21/10 15:40:36 RX: >RGP010100011416-0377074-038537290000003FFDB0000;ID=LN61;#767A;*26<

21/10 15:40:36 RX: >RGP010100011423-0377074-038537290000003FFDB0000;ID=LN61;#767B;*23<

21/10 15:40:37 RX: >RGP010100011432-0377074-038537290000003FFDB0000;ID=LN61;#767C;*22<

21/10 15:40:37 RX: >RGP010100011435-0377074-038537290000003FFDB0000;ID=LN61;#767D;*22<

21/10 15:40:37 RX: >GFFN_00004_EOF;ID=LN61;#767E;*11<

Onde:

  • O Virloc confirma o recebimento do comando respondendo GFFN_ACK;
  • Os registros RGP armazenados são entregues;
  • O Virloc envia um GFFN_00004_EOF, informando o fim da entrega e a quantidade de reportes entregues (4).

 

¶ Entrega Automática do LOG

Também é possível programar um evento que realize a descarga automaticamente, a partir da ação de um evento. Neste caso, é possível encaminhar a descarga de registros para o canal desejado, como GPRS ou Bluetooth.

Exemplos:

>SED001 IN07-- +- TRM AX {GFFNfTRM}< //Ao desligar a Ignição, descarrega o LOG via Serial TTL

>SED002 IN07-- +- GR0 AX {GFFNfGR0}< //Ao desligar a Ignição, descarrega o LOG via GPRS

>SED003 IN07-- +- BTT AX {GFFNfBTT}< //Ao desligar a Ignição, descarrega o LOG via Bluetooth

 

Dessa forma, os reportes serão encaminhados para o canal escolhido e descarregados da seguinte forma:

21/10 15:57:58 RX: >RAX003,;ID=LN61;#769B;*3D<

21/10 15:57:58 RX: >RGP010100013142-0377074-038537290000003FFDB0000;ID=LN61;#769B;*2D<

21/10 15:57:58 RX: >RGP010100013148-0377074-038537290000003FFDB0000;ID=LN61;#769C;*26<

21/10 15:57:58 RX: >RGP010100013151-0377074-038537290000003FFDB0000;ID=LN61;#769D;*29<

21/10 15:57:59 RX: >RGP010100013154-0377074-038537290000003FFDB0000;ID=LN61;#769E;*2D<

21/10 15:57:59 RX: >GFFN_00004_EOF;ID=LN61;#769F;*1C<

Onde:

  • O Virloc inicia enviando um reporte AX com o número do evento RAX003;
  • Os registros RGP armazenados são entregues;
  • O Virloc envia um GFFN_00004_EOF, informando o fim da entrega e a quantidade de reportes entregues (4).

 

¶ Filtro de Data/Hora

É possível especificar um período de tempo para entregar somente os registros correspondentes. Para isso utiliza-se o comando GFFT, que possui a seguinte sintaxe:

>GFFT010921000000300921235959<

Onde:

010921000000 = Data/hora inicial, no formato ddMMaahhmmss (ex.: 01/09/21 00:00:00);

300921235959 = Data/hora final, no formato ddMMaahhmmss (ex.: 30/09/21 23:59:59).

 

¶ ⛔ Cancelar Entrega de LOG

Para parar uma descarga de LOG em andamento, pode ser utilizado o comando:

>GFFC<

Retorno:

>GFFN_ACK<
