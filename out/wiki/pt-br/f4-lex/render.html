¶ ? Introdução - LEX

O LOG de Dados é uma memória Flash incorporada aos equipamentos VIRLOC 08 e VIRCOM 07 para salvar permanentemente os relatórios e dados que ficam à disposição, podendo trabalhar com a funcionalidade de “Caixa Preta”.

Nos equipamentos da Linha VIRLOC, existem três LOGs de dados divididos em partes de tamanhos configuráveis:

  • LOG1 - LOG
  • LOG2 - LG2
  • LOG Estendido - LEX

Para salvar mensagens em dados o LOG1, LOG2 ou LOG Estendido, deve ser feito através dos eventos utilizando como destino LOG, LG2 e LEX para cada um dos LOGs respectivamente.

Neste tópico serão abordadas as configurações e os usos do LOG Estendido (LEX). Para mais informações a respeito de LOG1 e LOG2, clique aqui.

 

A configuração de memória para LOGs e Buffers será realizada utilizando os comandos especificados no tópico abaixo em Configuração do Log.

O LOG Estendido permite o uso de diversos tipos de formato de mensagem, que estão detalhados em Reports configuráveis para Log Estendido.

As consultas/descargas dos dados LOG's serão realizadas através de qualquer um dos comandos especificados em Descarga do LOG.

As consultas podem ser filtradas para obter as informações realmente necessárias. Para fazer isso, use os Filtros e Modificadores para descarga do LOG.

 

⚠️  LEX não está disponível para o VIRLOC 12.

 

¶ ⚙️ Configurações e Consultas do Log Estendido

 

¶ ? Configuração de memória

A divisão e configuração da memória reservada para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh), LOG1, LOG2 e Log Estendido LEX pode ser configurada pelo comando VSLG.

Configuração:

>VSLG0000,1024,8192,0,FOTA,0,LEX8192,FF8192<

Onde:

0000 = Senha geral do equipamento (padrão: 0000);

1024 = Quantidade de reports para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh); Valores permitidos: 0, 512, 1024, 2048, 4096 e 8192;

8192 = Quantidade de reports para LOG1. Pode ser configurado com valores múltiplos de 256 (mínimo: 8192);

0 = Quantidade de reports para LOG2, não é um parâmetro configurável (aparecerá apenas na consulta VQLG), indica o restante de memória após a configuração de LOG1.

FOTA = Reservar memória ou não para atualizações de firmware OTA [FOTA = habilita | NOFOTA = desabilita];

0 = Habilita backup de reportes descartados pelo buffer FIFO/LIFO. Pode ser configurado como LOG (para backup em LOG1), LG2 (para backup em LOG2), FXT (para backup em SD card) ou 0 para desabilitar;

LEX8192 = Quantidade de reports para Log Estendido LEX. Pode ser configurado com valores múltiplos de 256;

FF8192= Quantidade de reports para RF_FIFO (buffer de mensagens para comunicação entre os dispositivos na rede lora-mesh). Pode ser configurado com valores múltiplos de 256.

 

Consulta:

>VQLG<

 

Retorno:

>VRLG____,1024,00008192,00385024,1(FOTA)__,0(GFX_NO_BAK),8192,8192<

Onde:

0000 = Senha geral do equipamento (padrão: 0000);

1024 = Quantidade de reports para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh); Valores permitidos: 0, 512, 1024, 2048, 4096 e 8192;

00008192 = Quantidade de reports para LOG1;

00385024 = Quantidade de reports para LOG2, não é um parâmetro configurável (aparecerá apenas na consulta VQLG), indica o restante de memória após a configuração de LOG1.

1(FOTA)__ = Memória para atualizações de firmware OTA [FOTA = habilita | NOFOTA = desabilita];

0(GFX_NO_BAK) = Backup de reportes descartados pelo buffer FIFO/LIFO. Pode ser configurado como LOG (para backup em LOG1), LG2 (para backup em LOG2), FXT (para backup em SD card) ou 0 para desabilitar;

8192 = Quantidade de reports para Log Estendido LEX. Pode ser configurado com valores múltiplos de 256;

8192 = Quantidade de reports para RF_FIFO (buffer de mensagens para comunicação entre os dispositivos na rede lora-mesh). Pode ser configurado com valores múltiplos de 256.

  

? As regiões (VSRM), que são utilizadas para áudios, cercar embarcadas e listas de motoristas fazem parte da memória reservada para LOG1. Assim deve-se considerar os pontos dedicados para as Regiões na configuração de memória.

 

¶ ? Consulta de estado dos ponteiros

Para consultar a quantidade de registros alocados para o LEX, a quantidade de reportes gerados e entregues, utilizar a consulta GFFP, que irá retornar além das informações do LEX, outras informações de buffer LoRa.

>GFFP000049,000049,016384,000000,000000,000000,008192,00,00,05,0,0,0;ID=LN72;#8137;*34<

Onde:

000049 = Ponteiro de escrita LEX

000049 = Ponteiro de entrega LEX

016384 = Quantidade total de registros LEX

000000 = Ponteiro de escrita RF_FIFO

000000= Ponteiro de entrega RF_FIFO

000000= Ponteiro não utilizado

008192 = Quantidade total de registros RF_FIFO

00 = Ponteiro de mensagens RF_HI novos

00 = Ponteiro de mensagens RF_HI entregados

05 = Ponteiro de registros RF_HI

0 = Estado máquina de entrega mensagens RF_FIFO

0 = Estado máquina de entrega no utilizada

0 = Estado máquina de entrega LEX

 

¶ ? Apagar mensagens do LEX

Para limpar os registros armazenados em LEX, pode ser enviado o comando:

>GFFE0000<

Onde:

 0000 = é a senha do equipamento. Este comando pode levar algum tempo para executar a limpeza, e após concluir irá responder:

>GFFE_ACK;ID=LN72;#81FF;*0C<

 

¶ ? Destino e Formatos de Mensagem para LEX

 

¶ Destino LOG Estendido - LEX

O LOG Estendido - LEX, foi criado na linha VIRLOC e VIRCOM à partir do firmware 152B03, assim otimizando o armazenamento em memória com os registros guardados em formato binário. O que otimiza mais ainda este espaço é a possibilidade de armazenar estes registros de forma incremental. No LEX é possível salvar os formatos de mensagem pré-definidos e também os reportes configuráveis com as opções de formato que serão detalhadas a seguir.

Assim foi criado o destino LEX para ser utilizado nos eventos dentro da programação XVM.

? Exemplo: 

>SED001 TD01++ +- LEX GP<

  • Cada registro LEX tem um comprimento de 8 bytes, onde 1 byte de status + 7 bytes de dados é armazenado.
  • As mensagens HEX, RRx, RAX, RGP, RTT, RAD, RMI, RMA, RAS, RUN e RUD são salvas em formato binário para economizar espaço na memória.

 

¶ ? Reports configuráveis para Log Estendido

 

¶ Introdução - Contextualização

Um report configurável UC (SUC00..16), configura os dados que serão gerados por um evento, assim quando ele é declarado na estrutura do evento pode gerar um RUV, RUS, etc.

Assim temos por exemplo para o SUC01:

>SUC01 QGP,4,12 QGP,16,17 QGP,46,2 QGPD,40,3 QTT,49,8 QGP,33,3 QGP,36,3 QAD06,7,5 Q4P,20,5 GMV,24,27<

 

¶ Ux - Report configurável

Quando declarado no evento:

 >SED002 SG01++ +- TRM U1<

Report gerado: 

>RUS01,250521161640,-2351127-04669309,01,000,24060818,000,028,1559,00000,V+0092 X+0006 Y-0005 Z-0091;ID=VL81;#0020;*4E<

Onde:

RUS01 = mensagem com tipo de report configurável 01;

250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091 = dados da mensagem.

 

? Da mesma forma pode ser consultado com >QUS01<

 

¶ Vx - Report configurável com número de evento

Quando declarado no evento: 

>SED002 SG01++ +- TRM V1<

Report gerado: 

>RUV0102,250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091;ID=VL81;#001F;*31<

Onde:

RUV01 = mensagem com tipo de report configurável 01 com número de evento;

02 = mensagem gerada pelo evento 02;

250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091 = dados da mensagem.

 

? Da mesma forma pode ser consultado com >QUV01<

 

¶ Nx - Report configurável numérico otimizado para LEX

Quando declarado no evento: 

>SED002 SG01++ +- TRM N1<

Report gerado: 

>RUN01,250521162649,-2351121,-4669301,2,0,24050620,0,28,1559,0,92,8,-6,-91,;ID=VL81;#0021;*6F<

Onde:

RUN01 = mensagem com tipo de report configurável 01 em formato decimal otimizado para LEX (sem zeros a esquerda e com todos os campos separados por vírgula);

                 Cada mensagem Nx pode ter até 16 variáveis ​​decimais de 61 bits (+1152921504606846975 .. -1152921504606846976)

                 Mensagens Nx só podem ter valores numéricos decimais na captura SUC final. Os outros valores são ignorados. Para usar QGP, foi criada a consulta QGPD, que expressa o valor das entradas e idade da posição em decimal. 

250521162649,-2351121,-4669301,2,0,24050620,0,28,1559,0,92,8,-6,-91 = dados da mensagem.

? Da mesma forma pode ser consultado com >QUN01<

 

¶ Lx - Report configurável numérico diferencial para LEX

Quando declarado no evento:

 >SED002 SG01++ +- TRM L1<

Report gerado: 

 >RUL01,13,-1,2,-1,0,-200,0,0,0,0,1,0,0,-1,;ID=VL81;#0025;*56<

Onde:

RUL01 = mensagem com tipo de report configurável 01 em formato decimal diferencial para LEX (entrega a diferença entre o último report Nx ou Lx entregue);

13,-1,2,-1,0,-200,0,0,0,0,1,0,0,-1 = dados da mensagem.

 

?Da mesma forma pode ser consultado com >QUL01<

A cada 10 reports Lx salvos em LEX, deve-se salvar um report Nx para sincronizar os diferenciais e agilizar a entrega de LOG estendido com GFFN. A entrega do LOG estendido somente será feita a partir do primeiro Nx encontrado na memória.

 

¶ ? Descarga do LOG

Abaixo temos um exemplo de uso do LOG estendido para desenvolvimento de uma função Caixa Preta utilizando o Virloc. Nesse exemplo, serão salvos os dados de data/hora, localização e velocidade a cada segundo com ignição ligada.

>SUC01 QGPD,4,12 QGPD,16,17 QCT10,7,10< //Report configurável

>STD000010< //Disparador TD configurado com 10s

>SED010 TD00++ IN07++ LEX N1< //Evento para salvar em LEX um report N1 a cada 10s

>SED011 TT00++ EV10--&&IN07++ LEX L1< //Evento para salvar em LEX um report L1 a cada segundo, se o evento 10 não for executado

Dessa forma, enquanto a ignição estiver ligada, os dados serão salvos no LOG estendido utilizando o formato diferencial a cada segundo, e a cada 10 segundos no formato completo. Ao realizar uma descarga, os dados serão reconstruídos e enviados conforme abaixo:

22/10 12:56:58 TX: >GFFN;ID=EST2;#80CE;*61< //Pedido de descarga do LEX

22/10 12:56:58 RX: >GFFN_ACK;ID=EST2;#80CE;*77< //Resposta ao comando GFFN

22/10 12:56:58 RX: >RUN01,221021155642,-2352503,-4667910,16,;ID=EST2;#0036;*08<

22/10 12:56:58 RX: >RUN01,221021155643,-2352503,-4667910,16,;ID=EST2;#0038;*07<

22/10 12:56:58 RX: >RUN01,221021155644,-2352503,-4667910,17,;ID=EST2;#0039;*00<

22/10 12:56:58 RX: >RUN01,221021155645,-2352503,-4667910,17,;ID=EST2;#0036;*0E<

22/10 12:56:58 RX: >RUN01,221021155646,-2352503,-4667910,17,;ID=EST2;#0037;*0C<

22/10 12:56:58 RX: >RUN01,221021155647,-2352503,-4667910,18,;ID=EST2;#0038;*0D<

22/10 12:56:58 RX: >RUN01,221021155648,-2352503,-4667910,18,;ID=EST2;#0039;*03<

22/10 12:56:58 RX: >RUN01,221021155649,-2352503,-4667910,18,;ID=EST2;#0037;*0C<

22/10 12:56:59 RX: >RUN01,221021155650,-2352503,-4667910,17,;ID=EST2;#0038;*04<

22/10 12:56:59 RX: >RUN01,221021155651,-2352503,-4667910,17,;ID=EST2;#0036;*0B<

22/10 12:56:59 RX: >GFFN_00010_EOF;ID=EST2;#003A;*60< //Fim da descarga de 00010 registros armazenados em LEX

A entrega dos reportes é feita para o mesmo canal de comunicação pelo qual foi recebido o GFFN.

 

¶ ? Cancelar descarga do LOG

Para cancelar a entrega do Log Estendido, utiliza-se o comando:

>GFFC<

Ao qual será retornada a confirmação:

>GFFC_ACK<

 

¶ ? Filtros e Modificadores para descarga do LOG

Conforme mostrado no exemplo acima, o comando GFFN solicita as novas mensagens não entregues do LOG estendido. Porém podem ser adicionados à solicitação modificadores para alterar parâmetros da entrega, e filtros para determinar que reportes específicos sejam entregues de acordo com a necessidade. Os modificadores e filtros podem ser combinados para refinar a busca.

 

¶ Redirecionamento de Entrega

Para descarga automática, podemos utilizar o comando GFFN na ação de um evento, encaminhando os reportes para um destino especificado. Para isso, utilizamos a letra ‘f’ após o GFFN e em seguida o destino desejado. Por exemplo:

>SED012 IN07-- +- SGN NN {GFFNfGR0}<

Onde:

f = Modificador para encaminhamento;

GR0 = Destino (comunicação GPRS).

Os destinos possíveis são:

  • GRx: via GPRS para o socket x;
  • WFx: via Wi-Fi para o socket x;
  • RFx: via LoRa para o socket x;
  • BTT: via Bluetooth;
  • TRM: via serial.

¶ VAR - Filtro de Variáveis

O filtro VAR permite realizar uma busca no log para que sejam entregues somente os reportes solicitados. Para construir o filtro, utiliza-se:

 

>GFFN_ALL VAR,hh,nn,min,max<

Onde:

hh = Hexadecimal do caractere separador (por exemplo, para separar por vírgulas, utiliza-se '2C');

nn = Quantidade de caracteres separadores a saltar;

min = Valor mínimo do filtro;

max = Valor máximo do filtro.

 

? Exemplo:

Tendo como base o reporte:

>RUN01,31221135954,-2352505,-4667904,4,;ID=LN72;#0020;*77<

Podemos montar o filtro:

>GFFN_ALL VAR,2C,4,4,9<

Onde:

2C = Caractere separador (2C = ‘,’)

4 = Quantidade de separadores a saltar separadores

4 = Valor mínimo;

9 = Valor máximo

 

? O modificador _ALL garante que a busca é realizada em todos os reportes. Caso seja omitido, a busca será feita apenas dentre os reportes não entregues.

¶ Modificador de cabeçalho

O cabeçalho padrão para os reportes do LOG estendido é RUNxx, mas pode ser alterado para um texto qualquer. Geralmente usado para manter compatibilidade no gateway com os reportes RUV. 

Além disso, também é possível definir o número de dígitos que o primeiro campo do reporte deve conter (tipicamente data/hora). Assim, datas como “09/12/21” são reportadas como “091221” e não "90121".

? Exemplo:

>GFFN_HDRRUV0195,12<

Onde:

_HDR = Modificador de cabeçalho;

RUV0195 = String que irá substituir o cabeçalho “RUNxx”;

12 = Quantidade de dígitos para formatar o primeiro campo do reporte (parâmetro opcional).

¶ NOF - Fim da descarga

Sempre que uma descarga do LEX é executada, ao final da entrega é enviada a mensagem EOF que pode ser vista nos exemplos anteriores. É possível anular o envio dessa mensagem com o parâmetro NOF.

? Exemplo:

>GFFNfGF0_NOF<

 

 

¶ 🆕 LEX - Novidades e otimizacoes (FW recentes)

Resumo das principais melhorias no LEX em firmwares recentes do VL08/VL12/VC07:

¶ ⚡ LEX otimizado para Vx (FW 156B30)

A partir do firmware 156B30 o LEX foi otimizado especificamente para entrega de mensagens via destino Vx (sem reconstrucao desnecessaria de pacotes). Aumenta a velocidade de descarga de LEX em conexoes lentas.

¶ 📦 Filtros GFFN no LEX (FW 154B10)

A partir do firmware 154B10 a consulta GFFN aceita filtros que otimizam a entrega:

  • _RUL (FW 153B13) - Entrega Lx sem reconstruir cabecalho (mais rapido)
  • _NOF (FW 153B25) - Omite EOF (end-of-file marker) na entrega
  • _HDRtext,Len (FW 154B01) - Substitui o cabecalho RUNxx por texto customizado
  • _VAR (FW 154B10) - Filtra apenas registros que contem variaveis
  • _RUL (FW 154B10) - Filtro de regras de entrega
  • _INF (FW 154B10) - Inclui campo INF nos registros
  • _HDR (FW 154B10) - Inclui cabecalho original
  • fXXX (FW 153B13) - Redireciona destino do LEX (XXX = indice do destino)

¶ 📡 Entrega LEX via Wi-Fi (FW 154P01)

A partir do firmware 154P01 a entrega da caixa preta LEX pode ser direcionada para Wi-Fi quando disponivel, evitando custo de dados GSM/4G em equipamentos com conectividade local (garagem, terminal, posto).

¶ 🌍 ZG0/ZG1 e LEX via Lora (FW 158B21)

A partir do firmware 158B21 os destinos ZG0 e ZG1 (gateways primario/secundario) podem entregar dados via Lora mesh, incluindo a caixa preta LEX. Util para frota off-grid com gateway central.

¶ 💡 LEX entrega _HI por padrao (FW 153B25)

A partir do firmware 153B25 a entrega via LEX usa o sufixo _HI (high priority) por padrao, garantindo que mensagens de LEX nao concorram com tracking normal e tenham prioridade na fila de saida.

¶ 📲 GFx via Bluetooth Classic/TRM (FW 158B21)

A partir do firmware 158B21 o LEX pode ser entregue via destino GFx direcionado a Bluetooth Classic ou a perifericos via TRM (terminal serial transparente), por exemplo descarga manual para um celular pareado.
