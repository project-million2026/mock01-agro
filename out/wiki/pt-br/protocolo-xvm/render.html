¶ PROTOCOLO XVM

 

A Linha VIRLOC utiliza como camada de transporte o protocolo UDP e o protocolo XVM (proprietário da Newtec) para garantir a integridade e entrega dos pacotes. Este sempre utiliza um início, um cabeçalho, parâmetros, ID do equipamento, um número de mensagens, checksum e caractere de finalização.

⚠️ Também é possível utilizar TCP nos modelos VL08, VC07, VL12 e VCone (indisponível apenas no VL06).

 

¶ Formato da Mensagem

 

Todos os pacotes enviados para e pelo dispositivo começam com o caractere de início > (maior) e terminam com <(menor) e um final de linha.

  • Em UDP, o final de linha são os caracteres 0x0D 0x0A
  • Em TCP, o final de linha é o caractere 0xC0

 

As mensagens no protocolo XVM são compostas de quatro partes as quais são delimitadas por ;(ponto e vírgula).
 

Exemplo de mensagem XVM:

>QGP;ID=1234;#8000;*5C<

As partes são as seguintes:

  • >  < Início e fim da mensagem
  • QGP = Conteúdo da mensagem
  • ID=1234 N° de ID do dispositivo
  • #8000 N° de mensagem
  • *5C Checksum
  • ; separadores

 

Todas as Mensagens começam com as letras S, Q, C, R, Gou Vdependendo da ação a ser seguida e continua com as letras do comando. Em seguida, segue uma série de caracteres variáveis que são os parâmetros do comando. A primeira letra pode ser:

 

 

¶ ID do Dispositivo

É uma cadeia de 4 caracteres alfanuméricos que representa o equipamento e permitirá reconhecê-lo dentro da frota. De fábrica é configurada uma ID que pode ser modificada pelo usuário. Às vezes você encontrará para usar algumas características especiais do equipamento você terá que incluir alguma letra atrás do número de identificação

 

¶ Número de Mensagem

 

É um número de formato hexadecimal de 16 bits (0000 para FFFF). Como a caixa de diálogo é bidirecional, as mensagens são numeradas para identificar que a resposta pertence a uma determinada mensagem enviada. Quando VL6 envia um relatório gerado por um evento, a mensagem irá transportar um número de 0001 a 7FFF e a base em resposta a essa mensagem deve enviar uma confirmação (ACK) com número de mensagem idêntico. Se o banco de dados (servidor) envia uma mensagem para VL6, a mensagem enviada deve ter um número de 8000 a FFFF e VL6 responderá a essa mensagem com outra mensagem conforme apropriado, mas ele tem o mesmo número de mensagem recebida.

Mensagem do equipamento: 0001 a 7FFF. 

Mensagem do servidor: 8000 a FFFF.

 

Exemplo do log de uma comunicação:
TX: >QSN;ID=5589;#8003;*50<
RX: >RSN20F75589_VL5;ID=5589;#8003;*51<
TX: >QVR;ID=5589;#8004;*4E<
RX:>RVR_VL6_FIRM6.12_IAP6_BR58_MDM:2G_FULLCAN_F91_SEC:0000;ID=5589;#8004;*6D<

 

¶ ✅ CheckSUM

É um número de 8 bits em formato hexadecimal calculado fazendo um XOR (ou- exclusivo) com todos os códigos ASCII dos caracteres que compõem a mensagem começando com > e terminando no último ; (ponto e vírgula incluído), mas não incluindo o * (asterisco) indicador do CheckSUM.

Calculo do Checksum:

Exemplo para calcular o checksum (código fonte em C)

 

/* get chksum */
int r;
unsigned char chksum;
chksum = 0;
for (r = 0; r < strlen(modem.rxbuff); r++) {
if ((modem.rxbuff[r] == '*') &&(modem.rxbuff[r-1] == ';')) break;
else
chksum = chksum ^ modem.rxbuff[r];
}

 

Exemplo em C#

 

public static string calculateChecksum(string data)
{
int r;
int calc = 0;
byte caracter;
string calculated_checksum; for (r = 0; r < data.Length; r++) {
if ((data[r] == '*') && (data[r-1] == ';')) break;
caracter = (byte)data[r];
calc = calc ^ (byte)caracter; }
calculated_checksum = calc.ToString("X"); return calculated_checksum;
}

 

A partir do módulo receptor de comunicação (servidor), se o checksum calculado coincide com o recebido e não é uma mensagem de resposta, você tem que confirmar a recepção correta respondendo a seguinte mensagem:

>ACK;ID=1017;#0093;*52<

Onde:

  • > início da mensagem
  • ACK identificador do tipo de mensagem
  • ; separador
  • ID=1017 ID do equipamento
  • ; separador
  • #0093 número da mensagem (igual ao recebido)
  • ; separador
  • *52 checksum desta mensagem
  • < fim da mensagem

 

Para determinar que não é uma mensagem de resposta é suficiente para comparar que o número de mensagem recebida é menor que 0x8000. Isto é porque as mensagens geradas a partir de VL6 tem um intervalo de 0x0000 a 0x7FFF e aqueles gerados a partir do módulo de comunicações (servidor) tem de ter um intervalo de valores de 0x8000 a 0xFFFF. Se você não confirmar a recepção, o VL6 vai enviar a mensagem até 5 vezes, ou seja, todas as mensagens devem ser respondidas com ACK.

Este protocolo é aquele usado por algum dos meios, assim que todas as mensagens emitidas a ele serão respondidas com o número correspondente da mensagem se os recebeu bem. Se a mensagem não for atendida, você deve enviar novamente até 3 vezes com um intervalo de 15 segundos entre cada tentativa.

 

¶ Múltiplas Mensagens 

Permite enviar múltiplos comandos em uma mesma mensagem do protocolo de transporte. O formato geral para este tipo de mensagem é: >comando1<>mensagem2<>mensagem3<>mensagemN;ID=1234;#8A1C;*XX<

Cada mensagem pode ter qualquer uma das mensagens definidas no protocolo de transporte VL6 entre ">" e "<".

Esse tipo de “múltiplas mensagens” pode ser enviado por qualquer mídia suportada pelo VL6. O tamanho máximo da mensagem depende dos meios de comunicação a serem usados, mas o máximo pode ser transportado 180 caracteres.

? Exemplo: 

>SSXP01<>SSXP11<>STD090025000010000600<>QTF;ID=1234;#FAEC;*XX<

 

¶ Comunicação com o servidor

Abaixo temos um exemplo de log da comunicação entre o VIRLOC e o servidor:

 

 

No envio da mensagem do VIRLOC para o servidor, temos um pacote do tipo RUV, que é o reporte configurável gerado por um evento.

>RUV0237,000,111,222,333,444,555;ID=0001;#167B;*33<

Onde:

  • RUV = Cabeçalho
  • 02 = Reporte do tipo 02 (cujo formato é definido pelo SUC02; consulte “Reportes Configuráveis” para mais detalhes)
  • 37 = Índice do evento que gerou o reporte
  • ,000,111,222,333,444,555 = Conteúdo separado por vírgulas, de acordo com o formato definido no reporte configurável

 

Em seguida, o servidor responde o ACK correspondente à mensagem, e o VIRLOC prossegue para o envio do próximo reporte na fila.
