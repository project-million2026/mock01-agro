Os dispositivos VIRLOC usam sockets UDP para transmitir mensagens encapsuladas com o protocolo proprietário XVM. Sempre contém um caractere inicial, cabeçalho, parâmetros, ID do equipamento, número da mensagem, checksum e um caractere final. Cada um desses elementos será detalhado nos tópicos a seguir, assim como o conteúdo dos reportes.

 

Para que o VL06 esteja pronto para funcionar conforme descrito nessa página, garanta que ele esteja programado com o script padrão, que está disponível no tópico Primeiros Passos.

 

¶ 1. Formato da Mensagem

Todas as mensagens enviadas pelo dispositivo começam com um caractere “maior” (>) e termina com um “menor” seguido de uma quebra de linha (<[CR][LF]) - valores em ASCII 0d13 e 0d10 ou 0x0D e 0x0A.

As linhas e reportes do script no Protocolo XVM são compostas de 4 partes separados um ponto e vírgula (;).

Exemplo de uma mensagem XVM:

>QGP;ID=1234;#8000;*5C<[CR][LF]

Onde:

As mensagens sempre começam com uma das seguintes letras S, Q, C, R, G ou V, dependendo da função a ser executada, e depois dela o comando ou mensagem em si. Opcionalmente, podem haver parâmetros na mensagem.

¶ 1.1 ID do Dispositivo 

O ID do dispositivo é um identificador de 4 dígitos alfanuméricos (A a Z e 0 a 9) que representa o dispositivo permitindo que o mesmo possa ser identificado na frota. O ID é configurado de fábrica como os 4 últimos dígitos do serial do VL06, mas pode ser alterado pelo usuário conforme necessário.

 

¶ 1.2 Número da Mensagem 

O número da mensagem é um hexadecimal de 16 bits (0000 a FFFF). Como a comunicação é bidirecional, o número da mensagem permite identificar qual mensagem é resposta de alguma outra.

Quando o Virloc envia um report para o servidor, a mensagem possuirá um número entre 0001 e 7FFF. O servidor deve então enviar uma mensagem de confirmação de recebimento com esse mesmo número de mensagem.

Se o servidor precisa enviar uma mensagem para o Virloc, ela deve conter um número de mensagem entre 8000 e FFFF, assim o dispositivo irá enviar a resposta ou confirmação do comando com esse mesmo número de mensagem.

Resumindo:

  • Mensagens do Virloc para o servidor: 0001 a 7FFF
  • Mensagens do servidor para o Virloc: 8000 a FFFF

Exemplo de um log de comunicação:

TX: >QSN;ID=5589;#8003;*50<

RX: >RSN20F75589_VL5;ID=5589;#8003;*51<

TX: >QVR;ID=5589;#8004;*4E<

RX: >RVR_VL6_FIRM6.12_IAP6_BR58_MDM:2G_FULLCAN_F91_SEC:0000;ID=5589;#8004;*6C<

TX: >QVR;ID=5589;#8005;*4F<

RX: >RVR_VL6_FIRM6.12_IAP6_BR58_MDM:2G_FULLCAN_F91_SEC:0000;ID=5589;#8005;*6D<

 

¶ 1.3 Checksum

O checksum é um hexadecimal de 8 bits calculado com uma operação XOR (OR exclusivo) entre todos os códigos da ASCII dos caracteres incluídos na mensagem desde o > até o último ponto e vírgula (;), mas não incluindo o asterisco que indica o checksum(*).

Exemplo para calcular o checksum em C

/* get chksum */
int r;
unsigned char chksum;
chksum = 0;
for (r = 0; r < strlen(modem.rxbuff); r++)
{
  if ((modem.rxbuff[r] == '*') &&(modem.rxbuff[r-1] == ';'))
    break;
  else
    chksum = chksum ^ modem.rxbuff[r];
}

 

Exemplo para calcular o checksum em C#

public static string calculateChecksum(string data)
{
  int r;
  int calc = 0;
  byte caracter;
  string calculated_checksum;
  for (r = 0; r < data.Length; r++)
  {
    if ((data[r] == '*') && (data[r-1] == ';'))
      break;
    caracter = (byte)data[r];
    calc = calc ^ (byte)caracter;
  }
  calculated_checksum = calc.ToString("X"); return calculated_checksum;
}

 

¶ 2. Confirmação 

No lado do servidor, sempre que o gateway recebe uma mensagem do dispositivo que não é uma resposta, e se ela possuir um checksum valido, é necessário mandar uma confirmação para o dispositivo com o seguinte formato:

>ACK;ID=1017;#0093;*52<[CR][LF]

Onde:

Para determinar se uma mensagem é uma resposta ou não, basta checar se o número da mensagem está abaixo de 8000, porque a mensagem gerada pelo Virloc estará entre 0000 até 7FFF. Qualquer número acima de 8000 é uma mensagem de resposta a um comando do servidor.

Se a mensagem gerada pelo Virloc não receber uma resposta ACK adequada, ele tentará reenviar até que seja recebido um ACK.

Quando o servidor manda uma mensagem e não recebe uma resposta do Virloc, ela pode ser reenviada com um intervalo de 15 segundos entre tentativas.

 

¶ 3. Mensagens Múltiplas

O protocolo permite que sejam enviados vários comandos dentro de uma mesma mensagem. O formato geral para esse tipo de mensagem é:

>comando1<>comando2<>comando3<>comandoN;ID=1234;#8A1C;*XX<

Cada um desses comandos pode conter qualquer instrução XVM entre “>” e “<”.

Este tipo de mensagem pode ser enviado a partir de qualquer comunicação suportada pelo Virloc, como UDP, Serial TTL ou Bluetooth. O maior tamanho dessa mensagem é de 180 caracteres, considerando todo o pacote (inclusive ID, número da mensagem e o checksum).

Por exemplo:

>SSXP01<>SSXP11<>STD090025000010000600;ID=1234;#FAEC;*XX<

 

¶ 4. Comunicação com o Servidor

Quando o Virloc se conecta com o servidor, a primeira mensagem que ele irá enviar é um reporte RGP. Este reporte contém informações básicas sobre o equipamento, mas não é disparado por um evento específico. Quando o servidor enviar um ACK adequado para este reporte, o Virloc irá começar a mandar os demais reportes normalmente.

Nós temos abaixo um exemplo de um log de comunicação entre o Virloc e o servidor:

Quando o Virloc envia uma mensagem disparada por um evento, ela será um reporte RUV, com um conteúdo variável. Ela terá um cabeçalho que possibilita a identificação do formato do conteúdo e qual foi o evento e disparou esse reporte. Veja Formatos de Reporte para mais detalhes.

Exemplo:

>RUV0237,000,111,222,333,444,555;ID=0001;#167B;*33<[CR][LF]

Onde:

Depois de receber essa mensagem, verificar que não é uma resposta e se o checksum for válido, o servidor deve responder com a seguinte mensagem de confirmação:

>ACK;ID=0001;#167B;*2C<

Então o Virloc continuará a mandar os próximos reportes.

¶ 6.2 Cadastro de Motoristas

O Virloc tem uma lista interna que pode armazenar os IDs de cartões RFID para a validação, quando se utiliza um leitor como periférico serial. Essa lista pode ser preenchida utilizando os comandos abaixo:

OBS: Para utilizar um leitor RFID é necessário conectar o TX (fio amarelo, OUT1) e RX (fio azul, IN03) na serial ou 1-wire do leitor. Também é necessário desativar a flag 08 com o comando SSH080.

 

Cadastro de IDs na lista:

>VSRT001,1644991399<

>VSRT002,1644991400<

>VSRT003,1644991401<

…

>VSRT999,1644991598<

 

Consultar um índice:

>VQRT001<

Resposta:

>VRRT0001,1644991399<

 

Deletar um ID de motorista:

>VSRT001,<

 

Onde:

  • 001 = índice da lista;
  • 1644991399 = Número do ID do motorista.

¶ 7. Solicitação de Reportes

É possível consultar “manualmente” cada um dos formatos de reporte existentes, com o intuito de obter as informações contidas sem que seja disparado um evento.

 

Para isso, deve ser enviada ao VIRLOC a mensagem:

>QUSxx<

Onde xx é o número do formato de reporte.

 

Exemplo:

>QUS00<

>RUS00,NT003,200422172106,04111337,05F12244,9.20_,BR59,V1.2_CAN,-,0,1,0,;ID=2244;#80C4;*5E<

 

No retorno, os campos terão o mesmo formato descrito no tópico “Formatos de Reporte”, com exceção apenas do campo RUV que passa a ser RUS e sem o número de evento.
