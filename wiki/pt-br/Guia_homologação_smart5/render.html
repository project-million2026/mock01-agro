¶ Homologação de Equipamentos ECO5, SMART5 e PRO5LITE

 

¶ Objetivo

Nessa página descreve todos os protocolos de comunicação utilizados pelos rastreadores e o servidor, servindo como principal referência para integração e homologação de novas plataformas de rastreamento.

 

¶ Escopo da Homologação

Durante a homologação é necessário validar que a plataforma seja capaz de:

  • Receber conexões TCP/IP dos equipamentos.
  • Identificar corretamente o dispositivo através do IMEI.
  • Receber e interpretar registros GNSS.
  • Confirmar o recebimento dos pacotes (ACK).
  • Decodificar eventos e entradas/saídas (IO).
  • Suportar comandos remotos utilizados pelo equipamento.
  • Validar atualização de configuração e firmware, quando aplicável.

¶ Principais etapas da comunicação

 

¶ 1. Identificação do equipamento

Ao conectar-se ao servidor, o equipamento envia seu IMEI, que deve ser validado pela plataforma.

Após a validação:

  • Servidor aceita ou rejeita a conexão.
  • Equipamento inicia o envio dos registros.

 

¶ 2. Recebimento dos registros

Cada registro enviado pelo equipamento contém informações como:

  • Timestamp
  • Latitude
  • Longitude
  • Velocidade
  • Altitude
  • Direção
  • Número de satélites
  • HDOP
  • Prioridade
  • Event ID
  • Dados de IO (sensores e periféricos)

Os registros podem ser enviados individualmente ou em lote.

 

¶ 3. Confirmação (ACK)

Após receber os registros, a plataforma deve responder informando:

  • quantidade de registros processados com sucesso

Caso contrário o equipamento retransmitirá os registros posteriormente.

Este é um dos principais testes durante a homologação.

¶ Estrutura dos registros

Cada registro possui dois blocos principais:

 

¶ Cabeçalho (Header)

Contém:

  • Timestamp
  • Prioridade
  • Coordenadas GNSS
  • Velocidade
  • HDOP
  • Event ID

 

¶ Corpo (Body)

Contém os parâmetros IO do equipamento:

  • Entradas digitais
  • Saídas digitais
  • Tensão
  • Ignição
  • Odômetro
  • Horímetro
  • CAN
  • OBD
  • Sensores
  • Informações específicas do modelo

¶ Eventos

O protocolo utiliza um Event ID para identificar o motivo da geração de cada registro.

Exemplos:

  • Ignição
  • Movimento
  • Excesso de velocidade
  • Entrada digital
  • Saída digital
  • Botão de pânico
  • Alimentação externa
  • Bateria
  • Geofence
  • Desligamento

A interpretação correta do Event ID é fundamental para que a plataforma gere os eventos equivalentes.

 

¶ Protocolos suportados

O documento descreve diversos protocolos utilizados pelos equipamentos.

 

¶ Protocolo básico

Envio de registros GNSS.

 

¶ Extended Protocol

Versão expandida do protocolo, permitindo transportar maior quantidade de informações de IO.

 

¶ FM & Server Protocol

Comunicação bidirecional entre servidor e equipamento para:

  • configuração
  • leitura de informações
  • envio de comandos
  • atualização de firmware

 

¶ Principais comandos servidor → equipamento

O protocolo define diversos comandos para gerenciamento remoto.

Entre os mais utilizados:

 

¶ Comandos SMS

Além da comunicação GPRS/TCP, os equipamentos aceitam comandos via SMS.

Exemplos:

  • info
  • version
  • imei
  • coords
  • getcfg
  • setcfg
  • getio
  • setio
  • getnetw
  • setnetw
  • reset
  • uptime

Esses comandos são úteis durante testes de homologação e manutenção em campo.

 

¶ MQTT

As versões mais recentes do protocolo incluem suporte à comunicação via MQTT utilizando mensagens JSON, oferecendo uma alternativa ao protocolo TCP tradicional.

 

¶ Configuração Remota

O protocolo permite:

  • Upload de configuração
  • Download da configuração atual
  • Alteração de parâmetros
  • Gerenciamento de perfis
  • Alteração de servidor
  • Configuração de rede

 

¶ Atualização de Firmware

O documento também descreve o processo completo de atualização OTA:

  • envio do pacote
  • transferência em blocos
  • confirmação
  • validação
  • reinicialização do equipamento
