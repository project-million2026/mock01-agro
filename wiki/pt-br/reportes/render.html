O Reporte é a mensagem que será enviada ao destino escolhido. O conteúdo desse pacote pode ter um formato definido ou configurável (em que é possível estabelecer quais informações serão informadas, seu formato e ordem conforme a necessidade da aplicação).

¶ Formatos definidos

São reportes que são pré-configurados no Virloc e fornecem informações num formato estabelecido que pode ser verificado no tópico de Consultas.

  • NN - Report nulo (sem report);
  • TT - Estrutura completa RTT (resposta da consulta QTT);
  • GP - Estrutura RGP (resposta da consulta QGP);
  • PF - Estrutura de dados GPS RPF (resposta da consulta QPF);
  • AD - Leituras AD do equipamento RAD (resposta da consulta QAD).

 

¶ Formatos Configuráveis

Um reporte configurável pode conter: (contando a partir do sinal de “>” até “<[CR][LF]”)

  • VL06: Até 133 caracteres para configuração e 140 para resposta do report.

   >SUC00 PLN#SSN,4,4 PLNODO:,4,4 QCT102,7,9 PLNHRMT:,4,5 QCT101,7,9 PLNV:,4,2 QCT109,7,3 PLNRPM:,4,4 QCT103,4,4 PLNTFL:,4,4 QCT15,7,10<

  >RUV0000,010100000433+0000000+000000000000000FF7B0000 00000000 000 01980000004800281520,010100000433+0000000+000000000000000FF7B0000 00000000 000 019;ID=2817;#03B3;*01<

  • VL06 MICRO F091: Até 133 caracteres para configuração e 224 para resposta do report.
  • VL8, VC7, VL12: até 203 caracteres

 

Configuração: 

>SUCxx CMD,from,len CMD,from,len CMD,from,len … <

  • xx - número do reporte configurável [00..15]
  • CMD - Comando a executar;
  • from - Capture a partir da posição de resposta ao comando;
  • len - Captura quantos caracteres (comprimento).

 

? Exemplo:

>SUC00 QTT,4,44 GIO,7,16 GVS00,7,10 GFQ,23,5 QCT18,7,10<

  • QTT: >RTT130120145322-2352432-046680250000003FF5A0000 04000000 000 00000000007004221313;ID=0000;#800E;*53<
  • GIO: >GIO00,0000000000000000;ID=0000;#800F;*05<
  • GVS00: >GVS00,0000000000;ID=0000;#8013;*72<
  • GFQ: >GFQ130120150154,00000,00000;ID=0000;#8014;*4E<
  • QCT18: >RCT18 2255911111;ID=0000;#8017;*7C<

>RUV0000,130120145322-2352432-046680250000003FF5A0000,0000000000000000,0000000000,00000,2255911111;ID=VL81;#0001;*3C<

 

Consulta da configuração:

>QUCxx<

  • xx = número do reporte configurável [00..15]

 

¶ Ux - Report configurável sem número de evento

Quando declarado no evento: >SED002 SG01++ +- TRM U1<

>RUS01,250521161640,-2351127-04669309,01,000,24060818,000,028,1559,00000,V+0092 X+0006 Y-0005 Z-0091;ID=VL81;#0020;*4E<

Onde:

RUS01 = mensagem com tipo de report configurável 01;

250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091 = dados da mensagem.

⚠️ Da mesma forma pode ser consultado com >QUS01<

 

¶ Vx - Report configurável com número de evento

Quando declarado no evento: >SED002 SG01++ +- TRM V1<

>RUV0102,250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091;ID=VL81;#001F;*31<

Onde:

RUV01 = mensagem com tipo de report configurável 01 com número de evento;

02 = mensagem gerada pelo evento 02;

250521160853,-2351129-04669318,01,000,24060718,000,028,1559,00000,V+0092 X+0007 Y-0005 Z-0091 = dados da mensagem.

⚠️ Da mesma forma pode ser consultado com >QUV01<

 

¶ Nx - Report configurável numérico otimizado para LEX

Quando declarado no evento: >SED002 SG01++ +- TRM N1<

>RUN01,250521162649,-2351121,-4669301,2,0,24050620,0,28,1559,0,92,8,-6,-91,;ID=VL81;#0021;*6F<

Onde:

RUN01 = mensagem com tipo de report configurável 01 em formato decimal otimizado para LEX (sem zeros a esquerda e com todos os campos separados por vírgula);

250521162649,-2351121,-4669301,2,0,24050620,0,28,1559,0,92,8,-6,-91 = dados da mensagem.

  • Cada mensagem Nx pode ter até 16 variáveis ​​decimais de 61 bits (+1152921504606846975 .. -1152921504606846976)
  • Mensagens Nx só podem ter valores numéricos decimais na captura SUC final. Os outros valores são ignorados. Para usar QGP, foi criada a consulta QGPD, que expressa o valor das entradas e idade da posição em decimal. 

⚠️ Da mesma forma pode ser consultado com >QUN01<

NÃO DISPONÍVEL PARA VL6

 

¶ Lx - Report configurável numérico diferencial para LEX

Quando declarado no evento: >SED002 SG01++ +- TRM L1<

>RUL01,13,-1,2,-1,0,-200,0,0,0,0,1,0,0,-1,;ID=VL81;#0025;*56<

Onde:

RUL01 = mensagem com tipo de report configurável 01 em formato decimal diferencial para LEX (entrega a diferença entre o último report Nx ou Lx entregue);

13,-1,2,-1,0,-200,0,0,0,0,1,0,0,-1 = dados da mensagem.

⚠️ Da mesma forma pode ser consultado com >QUL01<

 

  • A cada 10 reports Lx salvos em LEX, deve-se salvar um report Nx para sincronizar os diferenciais e agilizar a entrega de LOG estendido com GFFN. A entrega do LOG estendido somente será feita a partir do primeiro Nx encontrado na memória.

NÃO DISPONÍVEL PARA VL6

 

¶ AC - Mensagem de texto da ação de um evento

Quando declarado no evento: >SED002 SG01++ +- TRM AC {MENSAGEM<

MENSAGEM;ID=LN82;#3776;*3E<

Onde:

MENSAGEM = mensagem de texto definida na ação do evento.

 

¶ AX - Retorno de uma consulta da ação de um evento

Quando declarado no evento no VL6: >SED002 SG01++ +- TRM AX {QCT01,7,10}{QAD,44,4}<

Quando declarado no evento no VL8, VL12 ou VC7: >SED002 SG01++ +- TRM AX {QCT01,7,10 QAD,44,4}<

>RAX02,00153,1295;ID=LN82;#3778;*64<

RAX = mensagem do tipo AX;

02 = mensagem gerada pelo evento 02;

00153,1295 = dados da mensagem.

 

¶ UF - Complemento de reportes configuráveis anteriores

Quando declarado no evento no VL8, VL12 ou VC7: >SUF00 %%04 QSN,4,8<

Utilizando o report dentro de um evento: >SED03 SG01++ +- TRM A0<

>RUV0403,17L41700;ID=2222;#053B;*60<

04 = Índice do RUV a ser complementado;

03 = Índice do evento que o disparou ;

17L41700 = dados da mensagem.

 

 

¶ 🆕 Novidades em Reportes (FW acima de 156)

Recursos adicionados aos comandos de configuracao e composicao de reportes a partir do firmware 156:

¶ 📝 SUC opcao N - Sem virgulas separadoras (FW 156B28)

A partir do firmware 156B28 a configuracao de reportes SUC aceita o modificador ,N no final do comando, que faz com que o reporte seja gerado sem virgulas separadoras entre os campos (concatenacao direta). Util para parsing customizado no servidor.

Exemplo:

>SUC00,A,VCT01,VCT02,VCT03,N<

Gera reporte concatenado dos valores de CT01 + CT02 + CT03 sem virgulas. Sem o sufixo N o reporte teria virgulas entre os campos.

¶ 📋 SUF - Reporte complementar A0-A7 (FW 157B13)

A partir do firmware 157B13 o comando SUF permite criar reportes complementares SUF00 a SUF07 que sao mapeados aos formatos A0 a A7 nos eventos. Permite adicionar campos extras a reportes existentes sem modificar a estrutura principal.

Mapeamento:

  • SUF00 -> A0 (Complemento 0)
  • SUF01 -> A1 (Complemento 1)
  • SUF02 -> A2 (Complemento 2)
  • SUF03 -> A3 (Complemento 3)
  • SUF04 -> A4 (Complemento 4)
  • SUF05 -> A5 (Complemento 5)
  • SUF06 -> A6 (Complemento 6)
  • SUF07 -> A7 (Complemento 7)

Exemplo de uso em evento:

>SUF00,VCT15,VCT16<

>SED170 TT00++ +- ZG0 V0A0<

O evento envia V0 (formato configuravel 0) seguido de A0 (SUF00 = CT15 + CT16).

¶ 🆔 VSIP - Alterar primeira letra do ID (FW 157B13)

A partir do firmware 157B13 o comando VSIP aceita o parametro ID0x (onde x = 0-F) que altera a primeira letra do ID das mensagens enviadas para esse destino. Permite diferenciar mensagens por destino no servidor (ex.: V vs S vs T).

Exemplo (destino 0 com letra T como prefixo):

>VSIP0,1.2.3.4,12345,0,F,ID0T<

¶ 🔥 %UTxx em SUC - Estado de UT no reporte (FW 160B25)

A partir do firmware 160B25 o comando SUC aceita placeholders %UTxx (xx = 00 a 59) que sao substituidos pelo estado atual do UTxx (1 = disparado, 0 = nao). Util para enviar o estado de varios UTs em uma unica mensagem.

Exemplo:

>SUC00,A,%UT00,%UT01,%UT02,%UT03<

Gera reporte com o estado (0 ou 1) dos UT00 a UT03.

¶ 📨 RUSxx direto em SUC (FW 160B25)

A partir do firmware 160B25 e possivel referenciar diretamente respostas pre-formatadas RUSxx (resposta de UT) dentro de configuracoes SUC, permitindo composicoes mais ricas sem precisar de variaveis intermediarias.
