¶ Introdução - LOG1 e LOG2

 

¶ O que é o LOG de Dados

O LOG de Dados é uma memória Flash incorporada ao equipamento da linha VIRLOC F4 para salvar permanentemente os relatórios e dados que ficam à disposição, podendo trabalhar com a funcionalidade de “Caixa Preta”.

 

¶ Tipos de LOG disponíveis

Nos equipamentos da Linha VIRLOC, existem três LOGs de dados divididos em partes de tamanhos configuráveis:

  • LOG1 - LOG
  • LOG2 - LG2
  • LOG Estendido - LEX

 

¶ Como gravar dados em LOG

Para salvar mensagens em dados o LOG1, LOG2 ou LOG Estendido, deve ser feito através dos eventos utilizando como destino LOG, LG2 e LEX para cada um dos LOGs respectivamente.

 

? Neste tópico serão abordadas as configurações e os usos de respeito de LOG1 e LOG2. Para mais informações a respeito do  LOG Estendido (LEX), clique aqui.

 

A configuração de memória para LOGs e Buffers será realizada utilizando os comandos especificados no tópico abaixo em Configuração do Log.

O LOG Estendido permite o uso de diversos tipos de formato de mensagem, que estão detalhados em Reports configuráveis para Log Estendido.

As consultas/descargas dos dados LOG's serão realizadas através de qualquer um dos comandos especificados em Descarga do LOG.

As consultas podem ser filtradas para obter as informações realmente necessárias. Para fazer isso, use os Filtros e Modificadores para descarga do LOG.

 

¶ ⚙️ Configurações e Consultas do Log

 

¶ Configuração de memória

A divisão e configuração da memória reservada para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh), LOG1, LOG2 e Log Estendido LEX pode ser configurada pelo comando VSLG.

Configuração:

>VSLG0000,1024,8192,0,FOTA,0,LEX8192,FF8192<

Onde:

0000 = Senha geral do equipamento (padrão: 0000);

1024 = Quantidade de reports para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh); Valores permitidos: 0, 512, 1024, 2048, 4096, 8192, 16384 e 32768;

8192 = Quantidade de reports para LOG1. Pode ser configurado com valores múltiplos de 256 (mínimo: 8192);

0 = Quantidade de reports para LOG2, não é um parâmetro configurável (aparecerá apenas na consulta VQLG), indica o restante de memória após a configuração de LOG1.

FOTA = Reservar memória ou não para atualizações de firmware OTA [FOTA = habilita | NOFOTA = desabilita];

0 = Habilita backup de reportes descartados pelo buffer FIFO/LIFO. Pode ser configurado como LOG (para backup em LOG1), LG2 (para backup em LOG2), FXT (para backup em SD card) ou 0 para desabilitar;

LEX8192 = Quantidade de reports para Log Estendido LEX. Pode ser configurado com valores múltiplos de 256;

FF8192 = Quantidade de reports para RF_FIFO (buffer de mensagens para comunicação entre os dispositivos na rede lora-mesh). Pode ser configurado com valores múltiplos de 256.

 

Consulta:

>VQLG<

 

Retorno:

>VRLG____,1024,00008192,00385024,1(FOTA)__,0(GFX_NO_BAK),8192,8192<

Onde:

0000 = Senha geral do equipamento (padrão: 0000);

1024 = Quantidade de reports para Buffer FIFO/LIFO (GFx | WIFI | Lora-Mesh); Valores permitidos: 0, 512, 1024, 2048, 4096 e 8192;

00008192 = Quantidade de reports para LOG1;

00385024 = Quantidade de reports para LOG2, não é um parâmetro configurável (aparecerá apenas na consulta VQLG), indica o restante de memória após a configuração de LOG1.

1(FOTA)__ = Memória para atualizações de firmware OTA [FOTA = habilita | NOFOTA = desabilita];

0(GFX_NO_BAK) = Backup de reportes descartados pelo buffer FIFO/LIFO. Pode ser configurado como LOG (para backup em LOG1), LG2 (para backup em LOG2), FXT (para backup em SD card) ou 0 para desabilitar;

8192 = Quantidade de reports para Log Estendido LEX. Pode ser configurado com valores múltiplos de 256;

8192 = Quantidade de reports para RF_FIFO (buffer de mensagens para comunicação entre os dispositivos na rede lora-mesh). Pode ser configurado com valores múltiplos de 256.

 

⚠️ Observação importante sobre memória 

As regiões (VSRM), que são utilizadas para áudios, cercar embarcadas e listas de motoristas fazem parte da memória reservada para LOG1. Assim deve-se considerar os pontos dedicados para as Regiões na configuração de memória.

 

¶ Consulta de estado dos ponteiros

Para consultar a quantidade de registros alocados para os LOGs, a quantidade de reportes gerados e entregues, utilizar a consulta QDLP.

>RDLP INI00000000 PTR00000183 END00065536 LR00000108 / INI00065792 PTR00445952 END01834496 LR00445952 / SCAN00000000 SEND00000000;ID=LN72;#819B;*60<

Onde:

00000000 = Primeira posição de memória para LOG;

00000183 = Posição atual de escrita no LOG;

00065536 = Última posição de memória do LOG;

00000108 = Última posição entregue pelo LOG;

00065792 = Primeira posição de memória para LOG 2;

00445952 = Posição atual de escrita no LOG 2;

01834496 = Última posição de memória do LOG 2;

00445952 = Última posição entregue pelo LOG 2.

 

¶ Formatos de Mensagem para LOG

Abaixo está uma lista e um exemplo das possíveis respostas que podem ser gravadas / obtidas nos registros de LOG de dados.

 

¶ ?️ Report GP - Posição com data e hora

Quando um report do tipo GP é gravado no LOG de Dados, ele será formatado como:

>RGL110715030802-3597296-062735570000000FF5F25<

Onde:

RGL = Tipo de pacote GP armazenado em LOG;

110715030802 = Data no dia, mês, ano, hora, minuto, segundo (cada dado com 2 dígitos);

-3597296 = Latitude (se está carregada) ex: -35,97296°;

-06273557 = Longitude (se está carregada) ex: -062,73557°;

000 = Velocidade em Km/h;

000 = Orientação em graus (heading 0 ..359), Norte = 0, Este = 90, Sul = 180, Oeste = 270.

0 = Estado do posicionamento do GPS da última posição válida;

  • 0 - Posicionando com somente 1 satélite;
  • 1 - Posicionando com 2 satélites;
  • 2 - Posicionando com 3 satélites (2D);
  • 3 - Posicionando com 4 ou más satélites (3D);
  • 4 - Posição em 2D calculada por aproximação de quadrados mínimos;
  • 5 - Posição em 3D calculada por aproximação de quadrados mínimos;
  • 6 - Posição calculada sem satélites, por velocidade, sentido e tempo;
  • 7 - Sem detecção de movimento por acelerômetro, com a última posição válida de GPS.
  • 8 - Antena em curto circuito;
  • 9 - Antena não conectada;

FF = Quantidade de segundos desde que se tomou a última posição (Em Hexadecimal);

5F = Entradas digitais/Alimentação;

  • bit - 7 Ignição.
  • bit 6 - Fonte de Alimentação Principal.
  • bit 5  - Entrada digital 5
  • bit 4  - Entrada digital 4
  • bit 3  - Entrada digital 3.
  • bit 2  - Entrada digital 2.
  • bit 1  - Entrada digital 1.
  • bit 0  - Entrada digital 0.

25 = Número de evento gerado expressado em decimal.

? Diferente do formato RGP, o RGL não terá a última informação, de Diluição da Precisão Horizontal - HDOP.

 

Um exemplo de evento que gerará este report será:

>SED001 TT00++ SG01++ LOGGP<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

GP = As mensagens serão armazenadas no formato RGL.

 

Assim quando os reports forem descarregados terão o seguinte formato:

21/5 13:36:38 RX: >RGL210521161741-2350994-046688170000003005F01;ID=VL81;#0BDB;*5A<

21/5 13:36:38 RX: >RGL210521161735-2350997-046688140020003005F01;ID=VL81;#0BD5;*2C<

21/5 13:36:38 RX: >RGL210521161736-2350996-046688150020003005F01;ID=VL81;#0BD6;*2C<

21/5 13:36:38 RX: >RGL210521161737-2350996-046688150020003005F01;ID=VL81;#0BD7;*2C<

 

¶ ? Report M0 - Posição com data, hora e CT0

Quando um report do tipo M0 é gravado no LOG de Dados, ele será formatado como:

>LM0110715030802-3597296-06273557000065535025<

Onde:

LM0 = Tipo de pacote M0 armazenado em LOG;

110715030802 = Data no dia, mês, ano, hora, minuto, segundo (cada dado com 2 dígitos);

-3597296 = Latitude (se está carregada) ex: -35,97296°;

-06273557 = Longitude (se está carregada) ex: -062,73557°;

000 = Velocidade em Km/h;

0  = Posiçõs do GPS válida (0 = antiga) ;

65535 = Valor do CT00, com 16 bits [00000..65535]

025 = Número de evento gerado expressado em decimal.

 

Um exemplo de evento que gerará este report será:

>SED210 TT00++ +- SGN NN {SCT0 +1}<

>SED211 TT00++ +- LOGM0<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

M0 = As mensagens serão armazenadas no formato LM0.

 

Assim quando os reports forem descarregados terão o seguinte formato:

  >QDLN;ID=9905;#8C77;*7F<

 >RDL_ACK;ID=9905;#8C77;*24<

 >LM0070618171154-3460477-05862514000100004211;ID=9905;#5E9F;*19<

 >LM0070618171155-3460477-05862514000100005211;ID=9905;#5EA0;*17<

 >LM0070618171156-3460476-05862514000100006211;ID=9905;#5EA1;*17<

 >LM0070618171157-3460476-05862515000100007211;ID=9905;#5EA2;*15<

 >RDLF00004 LR00003922LOG;ID=9905;#5EA3;*49<

 

¶ ? Report GV - Estado do GPS

Quando um report do tipo GV, que contem as informações do GPS, é gravado no LOG de Dados, ele será formatado como:

>RGV03050720-2351179-0466929000120401<

Onde:

RGV = Tipo de pacote (resposta de uma consulta GV);

0 = Estado da antena GPS externa (quando presente no equipamento)

3 = Estado do posicionamento do GPS:

  • 0 - Sem Posicionamento.
  • 1 - Posicionando com só 1 satélite.
  • 2 - Posicionando com 2 satélites.
  • 3 - Posicionando com 3 satélites (2D).
  • 4 - Posicionando com 4 ou mais satélites (3D).
  • 5 - Posição em 2D calculada por aproximação de quadrados mínimos.
  • 6 - Posição em 3D calculada por aproximação de quadrados mínimos.
  • 7 - Posição calculada sem satélites (por velocidade, sentido e tempo).

05 = Número de satélites usados para fixar a posição GPS.

07 = Número de satélites vistos pelo GPS atualmente.

20 = Número de satélites que existem neste momento disponíveis em céu (calculado em função da data e posição atual).

-2351179 =  Latitude (sem filtro) ex: -23,51179°;

-04669290 = Longitude (sem filtro) ex: -046,69290°;

001 = Velocidade em Km/h;

204 = Orientação do GPS (heading em graus);

01 = Diluição da Precisão Horizontal - HDOP (0..50) da última posição válida;

 

¶ ? Report IN - Estado das entradas Digitais

Quando um report do tipo IN (como a resposta da consulta QIN), que contem as informações das entradas digitas do equipamento, é gravado no LOG de Dados, ele será formatado como:

>LIN00 010100062833 11011011<

Onde:

LIN = Tipo de pacote (estado das entradas digitais em LOG);

00 = Número do evento gerado;

010100 = dia, mês, ano.

062833 = Hora, minutos, segundos.

Estado das entradas:

  • 1 - Ignição;
  • 1 - Fonte de Alimentação Principal;
  • 0 - Entrada digital 5 (se disponível no equipamento);
  • 1 - Entrada digital 4 (se disponível no equipamento);
  • 1 - Entrada digital 3.
  • 0  - Entrada digital 2.
  • 1 - Entrada digital 1.
  • 1 - Entrada digital 0.

 

¶ ? Report AD - Estado das entradas Analógicas

>LAD00 07011601333102970000009404362620<

Onde:

LAD = Tipo de pacote (resposta de uma consulta AD em LOG);

00 = Número do evento gerado;

070116 = dia, mês, ano;

013331 = Hora, minutos, segundos;

0297 = Tensão entrada analógica 0;

0000 = Tensão entrada analógica 1;

0094 = Tensão entrada analógica 2;

0436 = Tensão da bateria interna;

2620 = Tensão da alimentação principal.

 

?Os valores de tensão são expressos em volts e possuem 4 dígitos, sendo dois inteiros e dois decimais. 

? Exemplo: 

0436 = 4,36 Volts.

 

Um exemplo de eventos que geraram este tipo de report será:

>SED002 SG02++ +- LOGAD<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

AD = As mensagens serão armazenadas no formato LAD;

 

Assim quando os reports forem descarregados terão o seguinte formato:

21/5 20:43:34 RX: >RDLF00001 LR00007606LOG;ID=VL81;#3B64;*25<

 

¶ ? Report AC - Texto Livre

Quando um report do tipo AC é gravado no LOG de Dados, ele será formatado como:

>LACXXddmmaahhmmssTexto<

Onde:

XX =  Número de evento que gerou o report em LOG;

ddmmaahhmmss  = Dia e hora do report;

  • dd - dois dígitos para a dia;
  • mm - dois dígitos para a mês;
  • aa - dois dígitos para a ano;
  • hh - dois dígitos para a hora (UTC, como no GPS);
  • mm - dois para os minutos;
  • ss - dois para os segundos;

Texto = Texto enviado no report (pode conter até 10 caracteres).

 

Um exemplo de eventos que geraram este tipo de report será:

>SED001 TT00++ SG01++ LOG GP<

>SED002 SG02++ +-LOGAC {INI_VIAGEM<

>SED003 SG02-- +- LOGAC {FIM_VIAGEM<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

AC = As mensagens serão armazenadas no formato de texto LAC;

INI_VIAGEM = Texto que será gravado.

 

Assim quando os reports forem descarregados terão o seguinte formato:

¶ ? Report AX - Retorno de uma Consulta

Quando um report do tipo AX é gravado no LOG de Dados, ele será formatado como:

>LAXxxddmmaahhmmssdados<

Onde:

xx =  Número de evento que gerou o report em LOG;

ddmmaahhmmss  = Dia e hora do report;

  • dd - dois dígitos para a dia;
  • mm - dois dígitos para a mês;
  • aa - dois dígitos para a ano;
  • hh - dois dígitos para a hora (UTC, como no GPS);
  • mm - dois para os minutos;
  • ss - dois para os segundos;

dados = dados da resposta RAX contidos na ação do evento (com limite para os 10 primeiros caracteres).

 

Um exemplo de evento que gerara este tipo de report será:

>SED002 SG02++ +- LOGAX {QSN,4,8}<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

V1 = As mensagens serão armazenadas no formato de texto LUV com os dados do SUC01;

QSN,4,8 = Consulta que gerará os dados para armazenar em log com report LAX.

 

21/5 20:29:47 RX: >RSN03L10001_VL10;ID=VL81;#8071;*7F< //Número de série do equipamento

21/5 20:29:16 TX: >QDLN;ID=VL81;#8070;*1D< //Comando para descarga do LOG

21/5 20:29:16 RX: >RDL_ACK;ID=VL81;#8070;*46<

21/5 20:29:16 RX: >LAX02 210521232911 03L10001;ID=VL81;#3B5E;*23<

 

¶ ? Report UC - Formato configurável

 Quando se utiliza um report configurável para armazenar os dados, estes terão o formato LUS (como resposta do QUS, sem o número do evento) ou LUV (com um RUV em log, com o número do evento gerado), ele será exibidos como:

 

>LUVxxyyddmmaahhmmss,dados<

Onde:

xx = número de reporte U/SUC (como RUS) ou V (como RUV);

yy = número de evento gerador;

ddmmaahhmmss  = Dia e hora do report;

  • dd - dois dígitos para a dia;
  • mm - dois dígitos para a mês;
  • aa - dois dígitos para a ano;
  • hh - dois dígitos para a hora (UTC, como no GPS);
  • mm - dois para os minutos;
  • ss - dois para os segundos;

dados = dados contidos no report (com limite para os 10 primeiros caracteres).

 

Um exemplo de evento que gerara este tipo de report será:

>SED002 SG02++ +- LOGV1<

Onde:

LOG = Destino LOG1 onde serão armazenadas as mensagens;

V1 = As mensagens serão armazenadas no formato de texto LUV com os dados do SUC01.

 

>SUC01QTT,33,3QCT01,8,4<

Onde:

01 = Repor configurável UC01;

QTT,33,3 = Consulta do dado de velocidade no TT;

QCT01,8,4 = Consulta da informação de RPM armazenada no CT01.

 

Assim quando os reports forem descarregados terão o seguinte formato:

>LUV0102 210521230436 017,1000;ID=VL81;#3B57;*24<

Onde:

01 = repor gerado com formato do SUC01;

02 = report gerado pelo evento 02;

210521230436 = data e hora em que o report foi gerado;

017 = velocidade do veículo;

1000 = RPM do veículo.

>QDLN<

>QDLNCT03<

03 = indica um dos CTs que contém o ponteiro onde o primeiro registro de LOG começará a ser pesquisado, no final da pesquisa, o ponteiro atual é salvo neste mesmo CT.

¶ ⏱️ QDLX - Consulta dos registros gerados nos últimos minutos

A consulta de todos os registros gerados nos últimos minutos à partir da hora atual pode ser realizada com o comando:

>QDLXaaaa<

Para LG2:

>QDLXaaaaLG2<

Onde:

aaaa  = Minutos à partir da hora atual.

 

¶ ? QDLY - Consulta dos registros gerados em um horário definido no dia atual

A consulta de todos os registros gerados em um intervalo de tempo definido dentro do dia atual pode ser realizada com o comando:

>QDLYhhmmsshhmmss<

Para LG2:

>QDLYhhmmsshhmmssLG2<

Onde:

hhmmss  = Hora do início dos reports para descarga;

  • hh - dois dígitos para a hora;
  • mm - dois para os minutos;
  • ss - dois para os segundos;

hhmmss  = Hora do final dos reports para descarga;

  • hh - dois dígitos para a hora;
  • mm - dois para os minutos;
  • ss - dois para os segundos;

 

¶ ? QDLT - Consulta dos registros gerados em um intervalo definido

A consulta de todos os registros gerados em um intervalo de tempo definido entre dia e hora pode ser realizada com o comando:

>QDLTddmmaahhmmssddmmaahhmmss<

Para LG2:

¶ ? QDLL - Consulta dos últimos registros

A consulta dos últimos registros gerados pode ser realizada com o comando:

>QDLLxxx<

Para LG2:

>QDLLxxxLG2<

Onde:

xxx = Número de registros para descarregar.

 

¶ ? QDLR - Consulta dos registros entre duas posições de memória

A consulta de todos os registros gerados em um intervalo de memória pode ser realizada com o comando:

>QDLRaaaaabbbbb<

Para LG2:

>QDLRaaaaabbbbbLG2<

Onde:

aaaaa = Número do primeiro registro a reportar [00000..32000];

bbbbb  Número do último registro a reportar [00000..32000].

 

¶ ? Consulta dos reports do LOG em formato Binário

O formato e os filtros para este comando são iguais aos do modo de texto, basta alterar o cabeçalho do comando de QDL para QLB.

? Por exemplo:

A consulta por intervalo definido - QDLT → >QDLT221002120000221002180000<    

para LG2                                           - QDLT → >QDLT221002120000221002180000LG2<

para LG2                                           - QLBT → >QLBT221002120000221002180000LG2<

 

¶ ? COPY - Consulta para copiar registros de um Log para outro

Acrescente copy ao comando QDL para copiar de um LOG para outro:

>QDLNcopy< 

  • Copia novos registros de LOG1 (LOG) para LOG2 (LG2)
  • Este comando interrompe qualquer entrega ativa do LOG

 

? Para enviar qualquer um desses relatórios automaticamente com os eventos, use o comando na ação do evento e, em seguida, use o filtro f e um destino (consulte Filtros e Modificadores para descarga do LOG).

 

¶ ? Cancelar uma consulta do LOG - QDLC

Para cancelar uma operação de descarga dos dados de log, a mesma pode ser realizada com o comando:

>QDLC<

Tem-se a opção de cancelar todos os relatórios de alta ou baixa prioridade, colocando um H ou um L, respectivamente, após a solicitação de cancelamento. 

Se nenhuma prioridade for indicada, ele cancela os relatórios de prioridade normal e baixa.

¶ Modificadores para envio de LOG via ar

 

¶ f - Redirecionamento de Entrega

Para descarga automática, podemos utilizar o comando QDLN na ação de um evento, encaminhando os reportes para um destino especificado. Para isso, utilizamos a letra ‘f’ após o QDLN e em seguida o destino desejado.

? Exemplo:

>SED012 IN07-- +- SGN NN {QDLNfGR0}<

Onde:

f = Modificador para encaminhamento;

GR0 = Destino (comunicação GPRS).

Os destinos possíveis são:

  • GRx: via GPRS para o socket x;
  • WFx: via Wi-Fi para o socket x;
  • RFx: via LoRa para o socket x;
  • BTT: via Bluetooth;
  • TRM: via serial.

 

¶ Z - Exigir confirmação

Formato: Z

Define que os registros das consultas sejam enviados um a um esperando a confirmação (ACK) para enviar o próximo. Desta forma evita-se perder registros.

? Exemplo: 

>QDLNZ<

 

¶ H - Prioridade Alta

Formato: H

Define os reportes como prioridade alta (HI). São enviados vários registros juntos, porém para cada um deve ser recebido um ACK. 

Caso não seja recebido, o reporte será reencaminhado até que o recebimento seja confirmado.

? Exemplo: 

>QDLNH<

 

¶ K - Prioridade Normal

Formato: K

Define os reportes como prioridade normal (OK). São enviados vários registros juntos, porém para cada um deve ser recebido um ACK. 

Caso não seja recebido, o reporte será reencaminhado pela quantidade de vezes configurada.

? Exemplo: 

>QDLNK<

 

¶ SL - Prioridade Super Baixa

Formato: SL

Solicita os reportes como prioridade super baixa (realiza uma única tentativa e descarta o reporte do buffer independente do recebimento de ACK).

? Exemplo: 

>QDLNSL<

 

? Por padrão, todas as respostas de consultas de LOG são enviadas em blocos e como prioridade baixa.
