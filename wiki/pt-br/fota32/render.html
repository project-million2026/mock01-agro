¶ Abertura do canal

Para inicializar o envio de firmware ou áudio, devem ser enviados os comandos >CRB< e >VSPG00500300< para o Virloc, com o protocolo XVM completo.

Para envio via UDP, essa etapa não é necessária.

Observações gerais:

  • O número de mensagem vai de ‘0x80000000’ a ‘0xFFFFFFFF’, e deve ser incrementado a cada envio de pacote.
  • Para todo byte que for igual a ‘0xDB’ dentro do arquivo SFB deve ser adicionado um byte ‘0xDD’ logo em seguida.
  • Todo byte que for igual a ‘0xC0’ dentro do arquivo SFB deve ser substituído pelos bytes ‘0xDBDC’ (isso é feito para não considerar os bytes ‘0xC0' que compõem o arquivo como final de pacote).
  • Manter somente o 0xC0 no final de cada pacote.

 

Observações para envio via UDP:

  • Não é necessário enviar o final de pacote (0xC0).
  • Nesse caso toda a lógica de substituição de bytes citada acima não é necessária.
  • Os pacotes terão um tamanho fixo de 544 bytes.

 

¶ Cálculo do Checksum

O checksum é calculado realizando um XOR de 4 em 4 bytes dos 540 bytes que o antecedem. Exemplo de código:

 

UINT32 BINA_getCRC(BYTE *message, UINT16 len)
{
     UINT16 x;
     UINT32 crc = 0;

     for (x = 0; x < len; x += 4)
              crc ^= get_hex32(&message[x]);
     crc = invert32(crc);

     return(crc);
}

// retorna um valor de 32 bits obtido da string de bytes
UINT32 get_hex32(BYTE *buff)
{
     UINT32 val;

     val = buff[0];
     val <<= 8;
     val |= buff[1];
     val <<= 8;
     val |= buff[2];
     val <<= 8;
     val |= buff[3];

     return(val);
}

UINT32 invert32(UINT32 val)
{
  BYTE x;
  UINT32 ret;

  ret = 0;
  for (x = 0; x < 4; x++)
  {
    ret <<= 8;
    ret |= (val & 0x000000FF);
    val >>= 8;
  }

  return(ret);
}

¶ Confirmação do Equipamento

  • Se o pacote enviado foi aceito pelo equipamento, ele responderá:

BINAVRFB + 0xFFFFFF22 + “_ACK” + SN + NDM + chksum + 0xC0

  • Se não foi aceito:

BINAVRFB + 0xFFFFFF22 + “_NAK” + SN + NDM + chksum + 0xC0

¶ Notas Adicionais

  • Após o envio do primeiro pacote de abertura de canal e a resposta de que foi aceito, o SFB deve começar a ser enviado imediatamente, pois deixar mais de 10 minutos entre os pacotes fará com que o equipamento entenda que o arquivo deixou de ser enviado, e o canal será fechado, tendo que abrir o canal novamente para continuar com o carregamento do SFB.
  • Se o dispositivo receber um NAK para algum pacote, indica que o pacote pode conter bytes corrompidos ou que o checksum foi calculado incorretamente pelo software emissor ou pode ser um firmware incorreto para o dispositivo. Nesse último caso será indicado como NAK_WF (wrong file).
  • Após o envio do último pacote, que ativa o SFB, se _NAK for recebido, indica que a integridade do arquivo recebido pelo dispositivo está corrompida e deve ser totalmente carregada novamente.
  • Após o envio do último pacote, que ativa o SFB, o dispositivo pode demorar até 1 minuto para responder novamente após indicar que aceitou a ativação do arquivo com _ACK.
  • Se não houver resposta a um pacote enviado ao dispositivo, ou se ele responder NAK, o pacote deve ser enviado novamente até 10 vezes com intervalo de 1 segundo entre os envios. Então, se o pacote não for aceito, o software deve abortar a transmissão e recomeçar ou notificar o usuário sobre o problema.
  • Para reconhecer o número de série do equipamento, o software pode enviar uma consulta QSN com protocolo XVM completo.
  • Todos os valores numéricos de 32 bits são enviados no formato LittleEndian, ou seja, o byte menos significativo é enviado primeiro.
